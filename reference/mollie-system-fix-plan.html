How to Fix Mollie: A Complete System Plan

Diagnosing the root causes of context loss, skimming, and wasted spend — and exactly what to change, what to keep, and how to validate.

Section 1

What's Actually Broken (Honest Diagnosis)

There are three separate problems happening at once. They look like one problem but have different fixes.

Problem 1: L1 (Always-Loaded) Files Are Too Heavy

Every session — including heartbeats and cron jobs — loads the following files before a single word is exchanged:

TOOLS.md122Bloated
USER.md69Trim possible

Total L1~659 linesToo much

Add today's + yesterday's memory files (~100–200 more lines) and Mollie is processing 750–850 lines of context before she reads your first message. That's where skimming comes from — not laziness, but volume.

Problem 2: Between-Session Promises Are Structurally Impossible

When a session ends, Mollie stops existing. There is no idle background process, no follow-through, no memory of intent. Every "I'll do X tomorrow" evaporates the moment you type /close. The only things that survive are written files. This has been generating broken promises and misplaced trust.

Problem 3: Daily Memory Has No TL;DR

When a fresh session loads memory/2026-04-19.md, it gets a raw log — chronological, dense, written in the moment. There's no signal at the top saying "here's the 3 most important things." So even when the information is there, it gets buried under the session log. That's why I had the Liverpool data in front of me and still fumbled it.

Section 2

How the System Is Supposed to Work (L1 / L2 / L3)

This is the architecture from the Reddit post you introduced in early April. It's the right model — it just hasn't been fully implemented.

LayerFilesWhen Loaded
L1 — Always Hot AGENTS.md, SOUL.md, USER.md, MEMORY.md, HEARTBEAT.md, TOOLS.md, IDENTITY.md Every single session, every message
L2 — On-Demand Warm Client context files, NOW.md, daily memory notes Loaded at session start or when topic comes up
L3 — Deep Reference Project files, reports, strategy docs, briefs, copy Only when actively working on something specific
The current problem: L1 is carrying L2 content. TOOLS.md has full integration runbooks (only needed when actually using an integration). AGENTS.md has explanatory paragraphs that belong in a reference doc. MEMORY.md has team roster detail that doesn't need to be hot every session. L1 should be dense pointers — not a library.

MEMORY.md 95 → target ~70 lines

Already lean from the April 2 reorg, but some sections can compress further:

  • Integrations list — remove individual key paths (they're in TOOLS.md). Keep account IDs only
  • Client index — one line per client: name + context.md path. Nothing else

Daily Memory Files — New TL;DR Convention New rule, no existing content changes

Every memory/YYYY-MM-DD.md file must start with a ## TL;DR block before any session log. 3 bullets maximum:

  • The most important thing we worked on
  • What's open / waiting
  • Anything critical not to miss (broken rule, hard decision made, etc.)

This is the anchor. It's the first thing I see in a new session — and it's designed to be impossible to skim past.

AGENTS.md — New Startup Synthesis Rule

Added to the startup sequence in AGENTS.md:

Before your first reply, verify three things from memory:
1. What was the most recent thing Michael and I worked on?
2. What is currently open or waiting?
3. Are any hard rules relevant to today's context?

If you cannot confirm all three, read the files again before responding.

AGENTS.md — New Between-Session Promise Rule

Added to AGENTS.md hard rules:

"I'll do X later" = queue it or don't say it.
Every end-of-session commitment must be immediately followed by one of:
(a) Write it to HEARTBEAT.md so the next heartbeat picks it up
(b) Create a cron job for it
(c) Do it right now before the session closes

If none of those happen, say explicitly: "I won't remember this unless we queue it. Should I?"
Never make a commitment that floats in conversation with no anchor.

Section 5

Validation Plan (No Broken Anything)

Before touching a single file:

  1. Backup all L1 files — copy current AGENTS.md, TOOLS.md, MEMORY.md, USER.md to reference/archive/pre-fix-2026-04-19/. One command, 30 seconds, full rollback available.

After changes are made:

  1. Start a fresh session — type /new and see what loads
  2. Liverpool test — ask Mollie about Liverpool without loading any client files. She should say "I need to pull the context file" — not guess from memory. That's correct behavior.
  3. Promise test — make an end-of-session commitment. Mollie must immediately queue it or explicitly flag it won't survive the close.
  4. TL;DR test — check that the next daily memory note has a TL;DR block at the top with 3 bullets.
  5. Cost test — run session_status after 2–3 days. L1 token load should drop measurably.
Nothing breaks. These are text file edits. No integrations, no Callie config, no heartbeat schedule, no cron jobs are touched. Worst case = a session feels lighter than expected and I need to be reminded of something I'd normally have memorized. Best case = sessions are faster, cheaper, and I stop skimming.
